# Security Policy

## 如何私下报告安全问题

如果你认为发现的是安全问题，请不要直接公开提交包含细节的 issue。

优先使用以下方式私下报告：

1. 如果仓库已启用 GitHub Security Advisories，请使用私密安全报告功能。
2. 如果尚未启用，请通过维护者在 GitHub 个人资料或仓库主页公开提供的联系渠道私下联系。
3. 如果以上方式暂时都不可用，请先提交一个不含漏洞细节的 issue，请求建立私下沟通渠道。

报告时建议包含：

- 影响范围
- 复现步骤
- 潜在风险
- 是否涉及敏感数据泄露
- 你建议的缓解方式（如有）

## 什么更适合作为普通 bug 报告

以下问题通常更适合走普通 issue：

- 页面样式或交互异常
- 场景文案不自然
- 评分不合理或反馈不准确
- 推荐表达不够贴切
- 本地启动失败或依赖安装问题
- 数据库初始化或迁移的普通错误

## 当前项目不应提交的敏感信息

请不要在 issue、PR、截图、日志或示例数据中提交以下内容：

- API key、token、cookie、密码
- 真实客户名称、公司内部信息、项目代号
- 个人隐私信息
- 本地数据库文件或其中的真实练习记录
- 本地环境路径、临时文件、编辑器缓存

## 当前安全边界说明

这个项目当前是一个本地运行的学习型 Web 应用，不是经过完整安全加固的生产系统。

当前仓库默认假设：

- 数据保存在本地 SQLite
- 不提供多用户鉴权
- 不默认暴露到公网

如果你计划把它部署到公网环境，请先自行评估：

- 部署方式
- 访问控制
- 数据存储与备份
- 反向代理与 HTTPS
- 日志与错误信息暴露
